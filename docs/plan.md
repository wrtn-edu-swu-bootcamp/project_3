# 배달앱 리뷰 텍스트 기반 재정렬 서비스 MVP 기획안

## 요약

배달앱의 별점 시스템은 인센티브와 편향으로 왜곡될 수 있어, 실제 만족도를 정확히 반영하지 못한다. 본 서비스는 사용자가 지역과 카테고리를 선택하면 배달의민족 상위 15~20개 가게의 리뷰를 자동 수집하고, 텍스트 내용을 분석하여 **"리뷰 내용 좋은 순"으로 재정렬**한다. 별점 4.8점 가게보다 4.5점 가게의 리뷰 내용이 더 좋을 수 있다는 인사이트를 제공하며, 토글 버튼으로 기존 별점순과 비교 가능하다. 2일 MVP는 독립형 웹 서비스로 구현되며, 크롤링부터 분석까지 5~7분 소요된다. 핵심 가치는 **별점 중심 정렬에서 리뷰 텍스트 중심 정렬로의 패러다임 전환**이다.

---

## 1. 서비스 개요

### 문제 정의
배달앱의 별점은 인센티브(리뷰 포인트), 보복성 저평가, 감정적 과대평가 등으로 왜곡되며, 정작 중요한 리뷰 텍스트 내용은 구매 의사결정에 충분히 반영되지 않는다. 기존 "별점 높은 순" 정렬은 실제 만족도와 불일치할 수 있다.

### 솔루션 요약
리뷰 텍스트를 감정 분석과 키워드 추출을 통해 평가하고, **가게를 "텍스트 점수 높은 순"으로 재정렬**하여 제공한다. 사용자는 별점이 아닌 실제 리뷰 내용 기반으로 가게를 비교할 수 있다.

### 타겟 사용자
배달앱을 사용하는 구매자 중, 별점만으로는 판단하기 어렵거나 리뷰 내용을 꼼꼼히 읽고 싶지만 시간이 부족한 사용자.

### 서비스 형태
독립형 웹 서비스 (모바일 웹 반응형 지원).

### MVP 핵심 가치
**"별점 대신 리뷰 내용으로 가게 순위 재정렬"** — 사용자는 지역과 카테고리를 선택하면, 자동으로 수집·분석된 결과를 텍스트 점수순으로 확인할 수 있다.

## 2. 시장 기회 분석

### 2.1 현재 문제점 상세화

- **별점 인센티브로 인한 과대평가**: 리뷰 작성 시 포인트를 주는 정책으로 인해 긍정 편향 발생.
- **악의적/보복성 리뷰**: 사소한 불만이나 감정적 이유로 과도하게 낮은 별점 부여.
- **리뷰 내용과 별점 불일치**: 별점 5점이지만 텍스트는 부정적이거나, 별점 3점이지만 내용은 긍정적인 경우 존재.
- **기존 정렬 방식의 한계**: "별점 높은 순"은 숫자만 반영하고, 리뷰 내용의 질과 신뢰도를 고려하지 않음.

### 2.2 타겟 플랫폼 비교

| 플랫폼 | 사용자 규모 | 검색 결과 접근성 | 크롤링 난이도 | MVP 지원 여부 |
|--------|-------------|------------------|---------------|---------------|
| 배달의민족 | 높음 (시장 1위) | 높음 (공개 검색 가능) | 낮음 (HTML 구조 단순) | ✅ 지원 |
| 쿠팡이츠 | 중간 | 중간 (일부 로그인 필요) | 중간 | ❌ 향후 확장 |
| 요기요 | 중간 | 중간 | 중간 | ❌ 향후 확장 |

**결론**: MVP는 배달의민족 1개 플랫폼만 지원하며, 접근성과 크롤링 용이성을 고려한 선택. 나머지 플랫폼은 향후 확장.

## 3. 핵심 기능 (MVP 범위)

### 3.1 지역 + 카테고리 기반 가게 자동 수집

**사용자 입력**
- 지역: 구 단위 드롭다운 선택 (예: 강남구, 서초구, 마포구 등)
- 카테고리: 미리 정의된 목록에서 선택
  - 치킨
  - 피자/양식
  - 중식
  - 한식
  - 일식/돈까스
  - 족발/보쌈
  - 야식
  - 분식
  - 카페/디저트
  - 기타 (배달의민족 주요 카테고리 기준)

**자동 수집 프로세스**
- 배달의민족 검색 결과 페이지 크롤링
- 상위 15~20개 가게 정보 수집 (가게명, 별점, 리뷰 URL)
- 각 가게별 리뷰 텍스트 수집 (최소 10개, 최대 50개)
- 수집 시간: 가게당 15~20초 예상, 총 5~7분 소요
- 프로그레스 바로 진행 상황 실시간 표시

### 3.2 리뷰 텍스트 기반 점수 산출 + 재정렬 (핵심)

**분석 방식**
- 수집된 리뷰에서 감정(긍정/부정)과 키워드(맛, 양, 배달, 포장 등) 추출
- **카테고리 2개로 단순화**: 
  - 음식 품질 (맛, 양, 신선도 등)
  - 배달/서비스 (배달 속도, 포장 상태, 친절도 등)

**결과 산출**
- 각 가게별 "텍스트 종합 점수" (0~5점) 산출
- 카테고리별 세부 점수 (0~5점)

**핵심 UI**
- **기본 정렬**: "텍스트 점수 높은 순"
- **토글 버튼**: `배민 별점순 ↔ 텍스트 점수순` 전환 (상단 고정)
- 각 가게 카드에 "배민 별점 vs 텍스트 점수" 나란히 비교 표시

### 3.3 신뢰도 경고 시스템

**경고 유형 2가지**
1. **불일치 경고**: 별점 평균 4.5+ 이지만 부정 키워드 비율 30% 이상
2. **표본 부족 경고**: 수집된 리뷰 수 10개 미만

**시각화**
- 경고 배지로 표시
  - "⚠️ 별점-리뷰 불일치"
  - "⚠️ 리뷰 수 부족"

### 3.4 출력 형식

**가게 리스트 화면**

상단: 정렬 토글 버튼 (고정)
- `[배민 별점순] ↔ [텍스트 점수순]`

가게 카드 (15~20개)
- **가게명** + 배민 별점 ★4.8 | 텍스트 점수 ★4.2
- 카테고리별 점수:
  - 음식 품질: ★4.5
  - 배달/서비스: ★3.8
- 강점 요약: "양이 많고 맛있다는 평가 많음"
- 약점 요약: "배달이 늦다는 리뷰 빈번"
- 신뢰도 경고 배지 (해당 시): ⚠️ 별점-리뷰 불일치

## 4. 사용자 시나리오 (User Journey)

**대표 사용자**: 직장인 김지수, 점심 메뉴를 고민 중. 배달앱의 별점만으로는 실제 맛집을 고르기 어려웠던 경험이 있음.

**단계별 사용 흐름**

1. **접속**: 서비스 접속 (점심 메뉴 고민)
   - 페인포인트: 배달앱 별점만으로 판단 어려움
   - 해결: 리뷰 내용 기반 재정렬 제공

2. **입력**: 지역 드롭다운에서 "강남구" 선택 + 카테고리 목록에서 "치킨" 선택
   - 페인포인트: 여러 앱을 비교해야 하는 번거로움, 오타 입력 가능성
   - 해결: 선택형 UI로 간편하게 입력, 자동 수집으로 한 번에 해결

3. **대기**: "분석 시작" 클릭 → 5~7분 대기
   - 페인포인트: 대기 시간 답답함
   - 해결: 프로그레스 바로 진행률 실시간 확인

4. **확인**: 가게 리스트 로드 (기본: 텍스트 점수순)
   - 페인포인트: 별점 높은 가게가 실제로 좋은지 의심
   - 해결: 텍스트 점수순으로 실제 만족도 반영

5. **비교**: 토글로 "배민 별점순"과 비교
   - 페인포인트: 순위 변화 이유를 모름
   - 해결: 카드에 강점/약점 요약 제공

6. **선택**: 경고 배지 확인 → 상위 2~3개 중 최종 선택
   - 페인포인트: 신뢰도 불확실
   - 해결: 불일치/표본부족 경고로 투명성 확보

7. **주문**: 배달앱에서 주문 (외부 연동)
   - 결과: 의사결정 시간 단축, 실패 주문 감소

## 5. 차별화 포인트

### 핵심 차별점: 별점 중심 → 텍스트 점수 중심 패러다임 전환

**Why This Matters**
- 기존: "별점 4.8점 가게가 최고다"
- 우리: "별점 4.8점이지만 리뷰 내용은 4.2점, 별점 4.5점이지만 리뷰 내용은 4.7점일 수 있다"
- **발견의 가치**: 숨겨진 진짜 맛집 발견, 과대평가된 가게 회피

### 세부 차별점

1. **사용자 편의 극대화**: 지역+카테고리를 드롭다운에서 선택하면 자동 수집·분석 (클릭 2번으로 모든 작업 완료)
2. **투명한 비교**: 토글 버튼으로 별점순 vs 텍스트 점수순 즉시 비교
3. **신뢰도 신호 제공**: 불일치/표본부족 경고로 판단 근거 투명화
4. **무료/간편**: 로그인 없이도 사용 가능

## 6. 기대 효과

### 구매자 관점

- **실패 주문 가능성 체감 20% 감소** (설문 기반 목표)
  - "주문했는데 기대 이하였다" 경험 감소
- **의사결정 시간 단축**: 평균 10분 → 3분 (설문)
  - 리뷰 일일이 읽지 않아도 요약 정보로 빠른 판단
- **주문 후 만족도 향상** (설문, 5점 척도 평균 4.0+ 목표)
  - 텍스트 점수 기반 선택이 실제 만족도와 더 일치

### 생태계 관점

- **리뷰 신뢰 회복**: 별점 인센티브/조작의 영향력 감소
- **진짜 맛집 발견 기회 증가**: 별점은 낮지만 실제로는 좋은 가게 조명
- **악의적 리뷰의 영향력 감소**: 텍스트 종합 분석으로 극단적 리뷰 희석

### 소상공인 관점 (확장 관점)

- 향후 사장님 리포트 기능 추가 시, 리뷰 기반 개선 포인트 제공 가능
- 별점 불리해도 리뷰 내용이 좋으면 상위 노출 기회

## 7. MVP 성공 지표 (KPI) — 2일 MVP 기준

### 기능 사용 지표

- **토글 사용률**: 별점순 ↔ 텍스트순 전환 ≥ 70%
  - 사용자가 비교 기능을 적극 활용하는지 측정
- **복수 검색 비율**: 2회 이상 지역+카테고리 검색 ≥ 50%
  - 재사용 의향 간접 측정

### 품질 지표

- **크롤링 성공률**: ≥ 85%
  - 가게당 최소 10개 리뷰 수집 성공률
- **평균 분석 시간**: ≤ 7분
  - 가게 15~20개 분석 완료까지 소요 시간
- **에러 없이 결과 도달률**: ≥ 90%
  - 입력부터 결과 확인까지 에러 없이 완료

### 만족 지표 (설문)

- **"주문 결정이 쉬웠다"**: 평균 4.0+ (5점 척도)
- **"텍스트 점수순 정렬이 유용했다"**: Yes ≥ 75%
- **"다시 쓸 의향"**: Yes ≥ 70%

### (선택) 운영 지표

- MAU, 일평균 검색 수 등은 MVP 이후 추적

## 8. MVP 범위 및 제약사항

### 포함

- ✅ 지역(구 단위) + 카테고리 선택 기반 자동 수집 (드롭다운 UI)
- ✅ 배달의민족 검색 결과 상위 15~20개 가게 크롤링
- ✅ 리뷰 텍스트 기반 점수 산출 (2개 카테고리: 음식 품질, 배달/서비스)
- ✅ 텍스트 점수순 ↔ 별점순 토글
- ✅ 불일치/표본부족 경고 (2개 유형)
- ✅ 웹 기반 (모바일 웹 반응형)

### 제외

- ❌ 로그인/회원가입
- ❌ 실시간 업데이트 (분석 1회만, 새로고침 시 재분석)
- ❌ 결제/주문 기능 (외부 배달앱 연동)
- ❌ 인센티브 탐지 (선택 기능으로 제외)
- ❌ 카테고리 세분화 (3개 이상)
- ❌ 지도/거리 기반 필터링

### 제약사항

- **크롤링 시간**: 가게 20개 분석 시 5~7분 소요 (프로그레스 바로 대기 완화)
- **HTML 구조 의존성**: 배달의민족 HTML 구조 변경 시 동작 불가 가능성
- **로그인 페이지 제약**: 로그인 필요 페이지는 수집 불가
- **IP 차단 가능성**: 과도한 요청 시 IP 차단 위험 (요청 간격 조절 필요)
- **지역 단위 제약**: "구" 단위 드롭다운만 제공 (예: 강남구 O, 역삼동 X → 향후 확장)
- **카테고리 제약**: 배달의민족 주요 카테고리만 제공 (9~10개 카테고리로 제한)

## 9. 향후 확장 가능성

- **지역 상세화**: "동" 단위 드롭다운 추가, GPS 기반 자동 선택
- **쿠팡이츠/요기요 지원 추가**: 다중 플랫폼 비교
- **플랫폼 API 연동**: 공식 API 허용 시 크롤링 대신 안정적 연동
- **브라우저 확장 프로그램**: 배달앱 페이지에서 원클릭 분석
- **필터링 추가**: 거리, 배달시간, 최소주문금액 필터
- **모바일 앱/PWA**: 네이티브 앱 경험 제공
- **사장님 대시보드(B2B)**: 리뷰 분석 리포트, 개선 포인트 제공
- **프리미엄 기능**: 광고 제거, 무제한 검색, 히스토리 저장

## 10. 리스크 & 대응

### 1. 크롤링 시간 지연

**리스크**
- 가게 20개 분석 시 5~7분 소요, 사용자 이탈 가능성

**대응**
- 프로그레스 바로 진행률 실시간 표시 (예: "15/20 가게 분석 중")
- 예상 완료 시간 표시 (예: "약 3분 남음")
- 대기 중 간단한 팁 문구 제공 (예: "알고 계셨나요? 별점과 리뷰 내용이 다른 경우가 30%나 됩니다")

### 2. 크롤링 차단 리스크

**리스크**
- 배달의민족이 크롤링을 감지하고 IP 차단 또는 CAPTCHA 발동

**대응**
- 요청 간격 조절 (가게당 2~3초 간격)
- User-Agent 헤더 설정 (일반 브라우저로 위장)
- MVP 이후 프록시/VPN 로테이션 고려
- 장기적으로 공식 API 연동 협의

### 3. 리뷰 편향 및 품질 문제

**리스크**
- 수집된 리뷰 자체가 편향되어 있거나 품질이 낮을 경우 (예: 인센티브 리뷰만 많음)

**대응**
- 불일치 경고로 인센티브 리뷰 의심 케이스 표시
- 표본 부족 경고로 리뷰 수 적은 가게 명시
- 설문에서 "분석 결과가 실제와 일치했는지" 피드백 수집 → 알고리즘 개선

### 4. 경고 오탐/과탐

**리스크**
- 불일치 경고가 너무 많거나 (과탐), 진짜 문제를 놓치거나 (오탐) 신뢰도 저하

**대응**
- MVP 단계: 불일치 기준을 보수적으로 설정 (별점 4.5+ & 부정 키워드 30% 이상으로 엄격하게)
- 사용자 피드백 수집: "이 경고가 도움이 되었나요?" 버튼
- 피드백 기반으로 임계값 조정 (예: 부정 키워드 비율 30% → 25%로 조정)
